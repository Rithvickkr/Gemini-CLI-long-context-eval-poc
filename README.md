# Long-Context Eval Dataset — Proof of Concept

> GSoC 2026 Proposal POC for [#23316](https://github.com/google-gemini/gemini-cli/issues/23316): Long-Context & Complex Reasoning Coding Eval Dataset

## What This Is

A working prototype of the curation pipeline and eval runner for building coding evaluation tasks that test LLM agents' ability to reason across large, multi-file codebases.

The core problem this benchmark tries to solve: most multi-file bug fixes are actually just retrieval. The agent finds the right function and patches it locally without ever needing to reason about why changing a contract in one module breaks an assumption three layers downstream. This POC focuses on building infrastructure that filters for tasks that genuinely require cross-component reasoning, not just cross-file token retrieval.

## What Is Verified vs What Is a Prototype

**Verified (built by hand, source read directly):**
- 8 tasks from google-gemini/gemini-cli and fastify/fastify. Every file path, line number, and reasoning chain was traced in the actual source at the pinned commit.
- The LongContextRig eval runner, RepoManager, schema validator, and contamination scorer.
- 4 bugs filed against gemini-cli while reading the source: #22170, #22171, #22164, #22159.

**Prototype (pipeline candidates, not manually verified line by line):**
- 20 tasks across express, koa, nestjs, typeorm, socket.io, vite, flask, gin, cobra. These were generated by the formulation pipeline using import graph analysis and GitHub issue cross-referencing. They pass schema validation and the reasoning chains are plausible, but they have not been hand-verified against merged fix PRs. In a real dataset these would go through a human review gate before being published.

This distinction matters. The point of the POC is to show the pipeline can find and structure real tasks at scale. The 8 verified tasks are the benchmark core. The 20 candidates show the curation approach works across different repos and languages.

## Architecture Decision: TestRig Extension vs Containerized

This POC uses a custom LongContextRig that extends the existing eval infrastructure rather than a separate containerized pipeline. Reasons:

- Results flow into runEval() and appear in the nightly aggregate summary
- Shares GEMINI_CLI_ACTIVITY_LOG_TARGET JSONL logging with the existing harness
- No Docker overhead per task
- Bare clone cache keyed on commit SHA avoids re-cloning the same repo across tasks
- Git worktree isolation gives each eval run a clean checkout without duplication

The tradeoff is that external repos require sparse checkout configuration per task. This is handled by RepoManager with the sparse_checkout_paths field in each task definition.

## Novel Metrics

### Reasoning Forcing Score (RFS)
```
RFS = (files_must_read - files_in_diff) x cross_references + backtracking
```
Measures how many cross-component boundary hops are required to find the fix, not just file count or token count. Tasks with low RFS are retrieval tasks in disguise and should be filtered out of the benchmark. Minimum enforced by schema: 3.

### Path Efficiency Score (PES)
```
PES = relevant_files_read / total_files_read
```
Measures how directly the agent navigated to the bug. High PES means the agent went straight to the right files. Low PES means it explored broadly but inefficiently.

### Item Discrimination Index (IDI)
```
IDI = P(pass | top_third_models) - P(pass | bottom_third_models)
```
Computed post-hoc from multi-model runs. Tasks where strong and weak models show the biggest gap are the most discriminating. Tasks with low IDI are not useful for the benchmark regardless of their complexity score.

## Failure Taxonomy (7 modes)

Rather than pass/fail, the LongContextRig classifies each result into one of:

| Mode | Description |
|------|-------------|
| `context_insufficient` | Agent read the right files but did not synthesize the cross-component relationship |
| `wrong_files_targeted` | Agent edited the wrong file entirely |
| `shallow_fix` | Agent patched the symptom but not the root cause |
| `cross_component_miss` | Agent fixed one component but missed the coupled change needed in another |
| `test_regression` | Fix broke existing tests |
| `timeout` | Agent did not complete within time limit |
| `complete_hallucination` | Agent invented a fix unrelated to the actual bug |

## Differential Testing (Catching Plausible-but-Wrong Patches)

A model's fix can pass tests but still be semantically wrong. The schema supports two fields to address this:

- **`weak_test_ids`**: Deliberately weak tests that a shallow fix would also pass. If the model's fix passes `weak_test_ids` but fails `fail_to_pass`, the rig classifies it as `plausible_but_wrong` rather than a simple failure. This catches fixes that address the symptom but not the root cause.

  Currently implemented on 4 tasks:
  - `gemini-cli-ignore-patterns-005`: weak test checks basic file ignoring; strong test checks directory-specific pattern matching
  - `flask-teardown-exception-swallow-001`: weak test checks teardown ran; strong test checks exception propagation chain
  - `flask-blueprint-url-prefix-merge-002`: weak test checks single-level prefix; strong test checks nested prefix composition under different registration orders
  - `cobra-persistent-flag-shadow-001`: weak test checks basic persistent flags; strong test checks shadow detection and value propagation

- **`semantic_correctness`**: Flags whether the test oracle alone is sufficient to verify correctness (`test_sufficient`), whether tests are necessary but not sufficient (`test_necessary_not_sufficient`), or whether manual review is required (`manual_review_required`). **Applied to all 28 tasks** — acknowledges the fundamental limitation that some fixes pass tests but are wrong, and makes this explicit in the schema rather than leaving it as an unstated assumption.

  Distribution: 4 tasks `test_necessary_not_sufficient` (have test oracles), 24 tasks `manual_review_required` (diff-only assertions).

The LongContextRig runs differential testing automatically when `weak_test_ids` are present in the task definition.

## How formulate.ts Works

```
git diff
  -> parse changed files
  -> build import graph (1-hop BFS from each changed file)
  -> expand files_must_read (importers + imported modules)
  -> compute RFS from boundary crossings
  -> scaffold symptom-based prompt
  -> generate hint_levels at 3 disclosure levels
  -> compute difficulty metrics
  -> output draft JSON (requires human review before publishing)
```

The curation pipeline treats a merged bug fix as the ground truth and works backwards. The diff tells you what changed. Import graph analysis tells you what a model needs to read to understand why. Human review confirms the reasoning chain is accurate.
